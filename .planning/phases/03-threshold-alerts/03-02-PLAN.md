<feature>
  <name>detectCrossings — pure per-metric crossing detection</name>
  <files>app/lib/alerts.ts, tests/alerts.detectCrossings.test.mjs</files>
  <behavior>
Signature:
```typescript
export type Crossing = { fullName: string; stars: number; metric: SubscriptionRow["metric"]; value: number; reason: string };
export function detectCrossings(
  sub: Pick<SubscriptionRow, "metric" | "threshold" | "window_days">,
  latestRepos: { fullName: string; stars: number }[],
  priorSnapshot: Map<string, number>,
): Crossing[];
```

Cases (write these as failing tests FIRST):
- **stars_abs**: threshold 1000. Repo at 1500 stars → crossing (value=1500, reason mentions "passed 1,000 stars"). Repo at 800 → no crossing.
- **stars_pct**: threshold 20 (percent), prior snapshot has `owner/x` at 100. Repo `owner/x` now at 130 → 30% growth ≥ 20 → crossing (reason "gained 30% ... crossing your 20% threshold"). Repo at 110 → 10% → no crossing.
- **stars_pct, no prior**: `owner/new` absent from priorSnapshot → cannot compute growth → NO crossing (avoid divide-by-zero / false fire). Assert empty.
- **velocity**: threshold 10 (stars/day), window_days 7, prior `owner/y` at 100. Repo `owner/y` now at 200 → (200-100)/7 ≈ 14.3 stars/day ≥ 10 → crossing. Repo gaining 50 over 7d → ~7.1/day → no crossing.
- **velocity, no prior**: no baseline → no crossing.
- Multiple repos in one call → returns one Crossing per repo that crossed, none for those that didn't.
  </behavior>
  <implementation>
After the tests are RED, implement `app/lib/alerts.ts`:
- `stars_abs`: `repo.stars >= sub.threshold`. value = repo.stars.
- `stars_pct`: prior = priorSnapshot.get(repo.fullName); if prior is undefined OR prior <= 0 → skip (no baseline → no false fire, Pitfall 2). growthPct = ((repo.stars - prior) / prior) * 100; crossing if growthPct >= sub.threshold. value = growthPct (rounded for the reason string).
- `velocity`: prior as above; skip if no baseline. perDay = (repo.stars - prior) / Math.max(1, sub.window_days); crossing if perDay >= sub.threshold. value = perDay.
- `reason` is a plain human string per CONTEXT "specifics" (e.g. `` `owner/x` gained 30% stars, crossing your 20% threshold ``). Do NOT build HTML here (escaping happens at the email layer, Plan 03/04). Pure data + a plain reason string only.
- Keep it I/O-free: no fetch, no db, no Date.now() inside (window_days is passed in; if a per-day calc needs elapsed time, derive it from window_days, not wall-clock — keeps the function deterministic and testable).
Commit RED then GREEN per the TDD cycle.
  </implementation>
</feature>
