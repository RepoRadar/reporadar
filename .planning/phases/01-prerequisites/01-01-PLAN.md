Export `sendEmail` with this EXACT signature (D-02):
```typescript
export async function sendEmail(opts: {
  to: string | string[];
  subject: string;
  html: string;
  text?: string;
  from?: string;
  replyTo?: string;
}): Promise<{ ok: boolean; id?: string; skipped?: boolean; status?: number; error?: string }>
```

Implementation rules:
- Read `const apiKey = process.env.RESEND_API_KEY;`. If unset/empty, `return { ok: false, skipped: true };` immediately — never throw, never fetch (D-04, mirrors translate.ts §38-39).
- `from` defaults to `opts.from || process.env.RESEND_FROM || "RepoRadar <onboarding@resend.dev>"` (D-03 — exact string).
- POST to `https://api.resend.com/emails` with headers `{ Authorization: \`Bearer ${apiKey}\`, "content-type": "application/json" }`.
- Build the JSON body from opts: `{ from, to: opts.to, subject: opts.subject, html: opts.html }`, plus `text` when `opts.text` is provided and `reply_to: opts.replyTo` (Resend's field name is `reply_to`) when `opts.replyTo` is provided. Do NOT send undefined fields.
- SECURITY (header-injection guard): `subject` MUST NOT contain CR or LF. Before sending, strip/reject newlines in subject: `const subject = opts.subject.replace(/[\r\n]+/g, " ").trim();`. This protects WS1/WS2 callers that will pass user-derived subjects later. (Resend takes a JSON body so SMTP header injection is not the vector, but normalizing subject newlines is cheap defense-in-depth and keeps the rendered subject clean.)
- On `res.ok`: parse JSON best-effort (`await res.json().catch(() => ({}))`), return `{ ok: true, id: parsed?.id }`.
- On non-ok: read `const t = await res.text().catch(() => "")`, `console.warn(\`[sendEmail] Resend ${res.status}: ${t.slice(0, 200)}\`)`, return `{ ok: false, status: res.status, error: t.slice(0, 200) }`. Never throw.
- Wrap the fetch in try/catch; on exception `console.warn(\`[sendEmail] exception: ${msg}\`)` and return `{ ok: false, error: msg }`. Never throw (D-04).

Do NOT add `runtime` exports — this is a lib, not a route.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller -> sendEmail() | WS1 contact form (Phase 2) and WS2 alerts (Phase 3) will pass user-derived `to`/`subject`/`html` into this lib |
| sendEmail() -> Resend API | RESEND_API_KEY (secret) crosses to api.resend.com over HTTPS |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | RESEND_API_KEY in sendEmail() | mitigate | Key read from `process.env.RESEND_API_KEY` only; never logged, never returned in the result object, never placed in HTML/response. On failure log only `res.status` + truncated body (200 chars), never the key. |
| T-01-02 | Tampering (header/subject injection) | sendEmail() `subject` from future user input | mitigate | Strip CR/LF from `subject` (`replace(/[\r\n]+/g, " ")`) before send. Body is JSON (not raw SMTP), so classic header injection is not reachable, but newline normalization is enforced as defense-in-depth for downstream untrusted callers. |
| T-01-03 | Tampering (HTML injection) | `html` body built by callers | mitigate | Export `escapeHtml` so all callers escape interpolated user values; deploy route already escapes repo/url/formFactor. Document that callers MUST escape untrusted interpolations (carried into Phase 2/3 plans). |
| T-01-04 | Denial of Service | sendEmail() outbound fetch hanging | accept | Resend is a fast managed API; no unbounded loop or retry added. Rate-limiting of callers is a Phase 2/3 route concern (contact form is rate-limited per CRED-03), not this lib's. |
| T-01-05 | Repudiation | failed sends silently lost | mitigate | Non-ok and exception paths `console.warn` with status/error so failures are traceable in Worker logs; result object surfaces `{ ok:false, status, error }` to callers for their own handling. |
</threat_model>
