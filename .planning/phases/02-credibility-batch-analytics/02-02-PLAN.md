   - `email`: required, basic shape check (must match `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`), max 200.

Build the HTML body using `escapeHtml()` on name/email/message (T-02-04: prevents HTML injection). Include a plain `text` alternative. Return 200 `{ ok:true, sent: result.ok }` (treat `skipped` as queued, not an error).
5. Wrap delivery in try/catch so a provider failure returns a graceful 200/queued result, never an unhandled 500.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| visitor (untrusted) → POST /api/contact | name/email/message are attacker-controllable |
| /api/contact → Resend (sendEmail) | user input flows into an outbound email |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Tampering (HTML/header injection) | /api/contact email body | mitigate | Apply `escapeHtml()` (Phase 1) to name/email/message before interpolating into HTML body. Subject CR/LF already stripped by `sendEmail`. `replyTo` is a validated email-shape field. |
| T-02-05 | Denial of Service (spam/abuse) | /api/contact | mitigate | In-memory fixed-window per-IP rate limiter (e.g. 5/60s) → 429 on exceed; field length caps (name 120, message 4000) bound payload size. |
| T-02-06 | Information Disclosure | error/log output | mitigate | Never log `RESEND_API_KEY` or full PII; sendEmail already redacts the key. Graceful 200/queued on missing config avoids leaking stack traces via 500s. |
| T-02-07 | Spoofing (forged sender) | reply-to header | accept | Reply-to is the user-claimed email; we do not verify ownership (v1). Low risk — it only sets where a reply goes, not auth. |
</threat_model>
